# v1.0.61（マウスレス配置抜き版）分離可否 見積もり調査（2026-07-01・隠密）

> 背景: 殿より「v1.0.7からマウスレス要素配置（キーボード配置モード）機能を抜いたv1.0.61を先にリリースできないか」との依頼。理由はマウスレス配置関連の不具合を含まない安定版を先に届けたいため。**実装・コミット操作は行っていない。分類・見積もり調査のみ。**

## 0. 先に伝えるべき重要な訂正

**「枠ラベル早期確定バグ＝マウスレス配置関連の不具合」という前提には疑義がある。** [keyboard-input-fix-plan.md](keyboard-input-fix-plan.md) ラウンド6〜8で特定した通り、この不具合の根本原因は **WinUI3自体の既知バグ（[Issue #6179](https://github.com/microsoft/microsoft-ui-xaml/issues/6179)）＋`RefreshPropertiesPanel`/`ScrollViewer`/`FocusSinkButton`周りの構造**であり、**枠のダブルクリック編集操作自体はマウスレス配置機能と無関係な一般機能**（v1.0.6以前から存在）。したがって、**マウスレス配置機能を除去しても、枠ラベルバグ自体が解消される保証はない**（症状の引き金にRootGridのKeyboardAccelerator群が関与する可能性はゼロではないが未検証）。除去する動機を「既知の不具合の回避」に置くなら、この点は殿・家老に必ず共有すべき前提としてここに明記する。

| **1. git revert連鎖** | 除去対象コミットを新しい順にrevert | — | **不可**。複合コミット（D群）は「一部のみ取り消す」ことができず、単純revertではB/E分類まで巻き込んで消してしまう。かつ後続コミットが同一ファイル・同一箇所を連続して触っているため、revert単体では高確率でコンフリクトが発生し機械的に完結しない |

## 3. 除去した場合の他機能への影響・リスク

- **ショートカットキー設定機能（167ef26）への影響**: なし。B分類はD群3コミットいずれからも独立して分離可能で、機能的な依存関係は無い。
- **一般キー処理（Escape/Backspace/PageUp/PageDown）への影響**: 上記手順3・4の通り「マウスレス配置判定の1行を消すだけ」で機能は保持できる想定。ただし実機での回帰確認は必須（`_keyboardModeActive`を参照する分岐を削った際の副作用がないか）。
- **154件(Core/Pdf)のxUnitテストへの影響**: なし。マウスレス配置・ショートカットキー設定ともにApp層のみでユニットテスト対象外のため、自動テストの安全網は薄く、実機確認への依存度が高い点は変わらない。
- **枠ラベルバグとの関係**: §0の通り、除去しても解消される保証はない。「安定版として届ける」動機を維持するなら、既知の制限事項として`docs/todo.md`に記載済みの内容はv1.0.61にもそのまま引き継がれる（もしくは、枠ラベルバグの発生条件を再検証しマウスレス配置有無での再現性を実機確認する追加調査が必要）。

## 4. 総合的な推奨コメント

- 方法2（cherry-pick＋手動分割）が目的に忠実だが3〜5人日規模。方法3（機能フラグ）は0.5人日未満だが目的を完全には達成しない。
- **§0の訂正が意思決定上もっとも重要**: 枠ラベルバグの回避が主目的なら、マウスレス配置を抜いても解消しない可能性が高く、費用対効果を再検討すべき。「マウスレス配置自体の不具合（Enter配置後の名前編集起動漏れ等、v1.0.7で既に修正済みのもの）を含まない、より枯れた状態で先に届けたい」という動機であれば方法2が妥当。
- 殿・家老には、まず§0の前提確認を仰いだ上で、除去の目的（安定版を早く届けたいのか／枠ラベルバグを避けたいのか）を明確にしてから方法を選定することを推奨する。
